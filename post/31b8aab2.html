MySQL 经典问题

01 基础架构：一条SQL查询语句是如何执行的

先聊聊MySQL的基础架构，

大致来说，MySQL可以分为 Server 层和存储引擎层两部分。

Server 层包括连接器、查询缓存、分析器、优化器、执行器等，涵盖了 MySQL 大多核心服务功能，以及所有的内置函数（如日期、时间、数学和加密函数等），所有跨存储引擎的功能都在这一层实现，比如存储过程，触发器、视图等。

而存储引擎层负责数据的存储和提取。其架构模式是插件式的，支持InnoDB、MyISAM、Memory等多个存储引擎，最常用的是InnoDB，它从MySQL5.5版本开始成为默认存储引擎。例如在 CREATE TABLE 语句中使用 ENGINE=MEMORY ，指定使用内存引擎创建表。

从图中不难看出，不同的存储引擎共用一个 Server 层，也就是从连接器到执行器的部分。

02 日志系统：一条SQL更新语句是如何执行的

执行语句前要先连接数据库，这属于连接器的工作。

前面说过，当表有更新时，跟这个表有关的查询缓存会失效，所以这条更新语句就会把表上所有换成结果都清空。

接下来分析器会通过词法和语法解析知道这是一条更新语句。优化器决定使用ID索引。执行器负责具体执行，找到这一行然后执行更新。

与查询流程不同，期间涉及到两个日志模块：redo log（重做日志） bin log（归档日志）

REDO LOG

WAL 技术：WAL 全称是 Write-AHead Logging，关键点是先写日志，在写磁盘。

具体来说，当有一条记录需要更新的时候，InnoDB引擎会先把记录写到 redo log 里面，并更新到内存，此时更新完成。同时，InnoDB会在适当的时候把操作记录更新到磁盘。

write pos 是当前记录的位置，一边写一边后移，循环往复。checkpoint 是当前要擦除的位置，也是往后推移并循环往复，擦除记录之前要把记录更新到数据文件。

有了 redo log，InnoDB就可以保证即使数据库发生异常重启，之前提交的记录也不会丢失，这个能力叫 crash-safe。
